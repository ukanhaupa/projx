# Changelog

All notable changes to projx are documented here. Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/); the project uses [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [1.9.1] - 2026-06-20

### Added

- **Per-instance ORM** — heterogeneous ORMs within one project ([#58](https://github.com/ukanhaupa/projx/issues/58)). Each backend instance can now run its own ORM instead of being locked to a single project-wide choice. `add <type> --name <dir> --orm <provider>` (and `add <type> --orm <provider>`) applies the ORM addon to **only** the newly added instance and records it in that instance's `.projx-component` marker; every other instance keeps its ORM. Resolution is family-aware (`resolveInstanceOrm`): instance marker → project-wide `.projx` orm when it is the same backend family → family default, so a Node global like `drizzle` never leaks into a Go/Rust/Laravel instance. The per-instance ORM flows through every generator (`setup.sh`, `ci.yml`, the per-instance Dockerfile migrate command) and `doctor`'s Go health check. Markers without an `orm` field fall back to the project-wide value (full back-compat); `update` preserves each instance's ORM. Verified across all four backend families — Node (prisma/drizzle/sequelize/typeorm), Go (gorm/sqlc/ent), Rust (seaorm), Laravel (eloquent) — including a single repo running drizzle + sequelize + gorm + sqlc side by side.

### Security

- **`fastify` + `express`** — pin `@opentelemetry/core` `>=2.8.0` via `pnpm.overrides` (pulled transitively through `@sentry/node`), clearing GHSA-8988-4f7v-96qf (moderate: unbounded memory allocation in W3C Baggage propagation) so each backend's `pnpm audit --prod` gate stays green. Mirrors the existing brace-expansion/qs override pattern.
- **`fastapi`** — bump `cryptography` 47→48.0.1 and `starlette` 1.0.1→1.3.1 (CVE-2026-48817 / 48818 / 54282 / 54283), and floor `msgpack` `>=1.2.1` via a uv constraint (transitive via `pip-audit` → `cachecontrol`; GHSA-6v7p-g79w-8964), clearing `pip-audit`. No `fastapi` bump required (it needs only `starlette>=0.46`); the 346-test suite stays green with the crypto module at 100% coverage.

## [1.9.0] - 2026-06-02

### Added

- **Go backend template** (`go/`) — Chi + GORM + entity registry + auto-routes, with JWT verifier (alg-confusion guarded), CORS, per-user rate limit (X-RateLimit headers), apperr → `{detail, request_id}` envelope, `/api/v1/_meta/schemas` sync emitter, distroless Dockerfile. `--auth=go` ports the full auth feature across **gorm/sqlc/ent** ORMs (signup / login / MFA TOTP+recovery / refresh rotation+replay detection / email verify / password reset / mailer / cron cleanup).
- **Rust backend template** (`rust/`) — Axum + SeaORM + tower middleware (`RequestId` task-local, CORS, logging, panic-recoverer), `ServiceConfig` (AES-256-GCM, NIST GCM TC13 wire-format compat with Node/Python/Go), per-user rate-limit, sync emitter, distroless runtime. `--auth=rust` lands the same auth surface.
- **Laravel (PHP 8.3+) backend template** (`laravel/`) — Laravel 11 + Sanctum + Eloquent + Pest + larastan, `RequestId` middleware, exception handler emitting `{detail, request_id}`, FrankenPHP-based Dockerfile, queue worker config, `service_configs` migration. `--auth=laravel` ports the auth feature with Eloquent.
- **Two new Go ORM addons**: `sqlc/` (typed-query generator) and `ent/` (schema-as-code). Each ships its own `internal/db/client.go`, entity-registry adapter preserving the GORM auto-routes contract (pagination / filter / search / order_by / soft-delete / lifecycle hooks / bulk ops), gen-entity templates, and Dockerfile/setup wiring. `addons/orms/_shared/` consolidates auto_routes + apperr/errors across sqlc + ent (single source of truth).

### Changed

- `applyOrmAddon` now supports `framework: 'go' | 'laravel' | 'rust'` in addition to the Node frameworks; Go addons can declare `gomodOverrides` (add / remove require entries) and ship a multi-stage Go Dockerfile (`golang:1.25-alpine` builder → `gcr.io/distroless/static-debian12:nonroot` runtime).
- CLI surface extended: `ORM_PROVIDERS` adds `gorm`, `sqlc`, `ent`, `seaorm`, `eloquent` with backend-family validation (`--orm=seaorm` requires `--components=rust`, etc.). `BACKEND_COMPONENTS` now lists `fastapi | fastify | express | go | rust | laravel`. `doctor` checks Go/Rust/PHP toolchain versions. `detect` recognizes Go (go.mod), Rust (Cargo.toml + axum), Laravel (composer.json + laravel/framework), sqlc (sqlc.yaml), ent (ent/schema/). `gen entity` dispatches per ORM (`appendGoEntity`, `appendSqlcEntity`, `appendEntEntity`, etc.).
- Templates (`ci.yml.ejs`, `setup.sh.ejs`, `docker-compose.yml.ejs`, `pre-commit.ejs`, `README.md.ejs`) and repo-level CI (`.github/workflows/ci.yml`, `scripts/ci-local.sh`, `.githooks/pre-commit`) all carry rust + laravel + go (with all 3 ORMs) sections.

### Security

- Go sqlc auth port: JWT verifier pins `*jwt.SigningMethodHMAC` type + `jwt.WithValidMethods()` to match gorm/ent — closes the RS256-vs-HS256 alg-confusion vector.
- Laravel auth port: removed `if (env('APP_ENV') !== 'production') $response['reset_token'] = …` branch that leaked raw password-reset tokens in non-prod responses. Tokens travel only via the email channel.

## [1.8.1] - 2026-06-14

### Added

- **New `nextjs` frontend component** ([nextjs/](nextjs/), [cli/src/utils.ts](cli/src/utils.ts)) — [#66](https://github.com/ukanhaupa/projx/issues/66). A React frontend on **Next.js (App Router)** — the house default for web frontends — at feature parity with [vitejs/](vitejs/) but idiomatic to App Router. Ships the same runtime surface: typed API client with `{ detail, request_id }` error parsing and `request_id` propagation (`ApiError.requestId`), request-path-only token refresh with a single shared in-flight `refreshPromise` lock (no `setTimeout` proactive refresh), DB-backed runtime config consumed at request time via a server-injected `window.__RUNTIME_CONFIG__` (env is bootstrap-only, no `NODE_ENV` branching), centralized error UI through `ErrorScaffold` wired into route-level `error.tsx`/`global-error.tsx`/`not-found.tsx`, edge `middleware.ts` auth gating, light+dark theming with a flash-free pre-hydration theme script (tokens only, all breakpoints), Toast/ConfirmDialog/Layout, and `@sentry/nextjs` (DSN-from-env gated). Built with `output: 'standalone'` + a multi-stage `Dockerfile.ejs`, security headers via `next.config.ts` `headers()`, and SWC-based E2E coverage instrumentation (`swc-plugin-coverage-instrument`, gated on `NEXT_COVERAGE`) that populates `window.__coverage__`. All source under `src/` (`src/app`, `src/lib`, `src/components`, `src/middleware.ts`), all tests under `tests/` mirroring `src/`. Own gates green: prettier, eslint (flat config), `tsc --noEmit`, `next build`, and vitest with v8 coverage at 96% statements / 92% branches / 95% functions / 98% lines (≥80% floor). **Fully selectable from the CLI**: `--components nextjs` (and `add nextjs`) scaffold it, `detect`/`init` recognize an existing `nextjs/` dir (by its `next` dependency, distinct from `vitejs`), and the generators emit its Docker Compose service (a standalone `node server.js` on port 3000 — no nginx/certbot, unlike `vitejs`), a path-filtered GitHub Actions job (format + lint + typecheck + build + test + audit), `setup.sh`/pre-commit blocks, and README entries. Wired into `scripts/ci-local.sh` (`nextjs` section) and the scaffold fuzzer. `vitejs` remains the default frontend.

### Changed

- **Renamed the `frontend` component to `vitejs`** ([vitejs/](vitejs/), [cli/src/utils.ts](cli/src/utils.ts)) — [#65](https://github.com/ukanhaupa/projx/issues/65). The React + Vite template directory and component id are now `vitejs`, freeing `nextjs` for a future template. `frontend` remains a permanent, supported **deprecated alias**: it is accepted at every input boundary (`--components frontend`, `add frontend`, `--<feature>=frontend`, and existing `.projx-component` markers of `type: frontend`) and normalized to `vitejs` via a single `normalizeComponent` helper. Existing scaffolded projects keep working under `add`/`update`/`init`/`doctor` with no manual migration; `update`/`add` migrate a legacy `frontend` marker to the canonical `vitejs` id in place. Display name is now "React + Vite".

### Security

- **`nextjs` pins a patched `postcss`** ([nextjs/package.json](nextjs/package.json)) — a `pnpm.overrides` entry (`postcss@<8.5.10` → `>=8.5.10`) lifts the transitive `postcss` pulled in by `next` past GHSA-qx2v-qp2m-jg93 (XSS via unescaped `</style>`), keeping the component's `pnpm audit --prod` gate clean. Mirrors the existing `fastify`/`express` override pattern.
- **Explicit, write-gated decryption of encrypted `service_configs` in the admin-panel** ([admin-panel/](admin-panel/)) — [#64](https://github.com/ukanhaupa/projx/issues/64). The encrypted `public.service_configs.config` cell now renders **masked** and is never auto-decrypted on page load. A per-value **Decrypt** action is shown **only** to admins whose session is currently in write mode; clicking it issues an HTMX request that re-checks write permission **server-side**, AES-256-GCM-decrypts the value, and reveals the plaintext inline over a `no-store` response. Read-only admins (and write-capable admins not currently in write mode) get a **403** even on a hand-crafted request — the gate is server-side, not hidden UI — and plaintext is never logged or written to the audit trail. The decryptor matches the backend writer's format exactly (`fastify/src/lib/crypto.ts`): `base64(IV[12] ‖ TAG[16] ‖ CT)`, AES-256-GCM, key from the new bootstrap-only `CRED_ENCRYPTION_KEY` env (base64 decoding to exactly 32 bytes). When the key is unconfigured the action never renders and the route returns a clear 503 — decryption is simply unavailable, no crash. Adds the `internal/secret` package (with a hardcoded Node-format cross-compat fixture test) and scopes the affordance to the `service_configs.config` cell only.
- **Mandatory TOTP 2FA for the admin-panel** ([admin-panel/](admin-panel/)) — [#63](https://github.com/ukanhaupa/projx/issues/63). The admin panel now enforces two-factor authentication on every admin. A password-only login no longer grants access to any admin route — the session is held in a forced enrolment flow (`/2fa/enroll`) until the admin scans a generated TOTP secret and verifies a first code. Enrolment issues ten single-use recovery codes (shown once, stored argon2id-hashed). After enrolment every login must pass a TOTP challenge (`/2fa`) with a valid code or recovery code before access is granted. The gate is server-side middleware (`admin_sessions.mfa_passed` + `admin_users.totp_enrolled_at`), not hidden UI; recovery codes are consumed atomically so they cannot be replayed. The TOTP secret is encrypted at rest with AES-256-GCM keyed off `SESSION_SECRET` (no plaintext, no new env). The login TOTP/recovery-code challenge is rate-limited per admin (NIST 800-63B): five consecutive failed verifications lock the account for 15 minutes via DB-backed `admin_users.mfa_failed_attempts`/`mfa_locked_until` columns (read-modify-write in a single atomic `UPDATE` so concurrent attempts across instances can't lose the count). While locked, every attempt is rejected with a non-revealing "too many attempts" error — even a correct code — and a successful verification resets the counter. Adds migration `0002_mfa.sql` (`admin_users.totp_secret_enc`/`totp_enrolled_at`/`mfa_failed_attempts`/`mfa_locked_until`, `admin_sessions.mfa_passed`, `admin_panel.admin_recovery_codes`) and the `github.com/pquerna/otp` dependency. The bootstrap admin is not pre-enrolled — it goes through the same forced flow on first login.

### Fixed

- **`add <component> --auth <target>` applies features when the component already exists** ([cli/src/add.ts](cli/src/add.ts)) — [#56](https://github.com/ukanhaupa/projx/issues/56). `add` exited early with "Nothing new to add" whenever every requested component was already present, before reaching `applyFeatures` — so `add fastify --auth fastify` on a project that already had fastify silently did nothing. The early exit now only fires when there is nothing to add **and** no features were requested; otherwise `add` downloads the repo, resolves the existing instances, and applies the requested features against them (ORM-validated, idempotent via the marker's `features` field).

## [1.7.7] - 2026-06-10

### Security

- **Refresh-token rotation TOCTOU race fixed across all auth ports** ([features/auth/](features/auth/)) — `/auth/refresh` previously read the token row, checked `rotated_to`/`revoked_at`, and only then rotated, so two concurrent refreshes with the same token could both pass the check and both receive new sessions. Rotation now starts with a guarded claim (`UPDATE … WHERE id = ? AND rotated_to IS NULL AND revoked_at IS NULL`); a claim that affects zero rows means the caller lost the race and goes through the replay-detection path (session tokens revoked, `replay_detected_at` stamped, 401 `token_replay_detected`). Applied to all nine ports — fastify + express × prisma/drizzle/sequelize/typeorm and fastapi — each with a new concurrency test firing 8 simultaneous refreshes and asserting exactly one winner.

### Changed

- **Backend Sentry config moved from env to `service_configs`** ([fastify/src/lib/sentry.ts](fastify/src/lib/sentry.ts), [express/src/lib/sentry.ts](express/src/lib/sentry.ts)) — the Node backends no longer read `SENTRY_DSN`/`SENTRY_ENVIRONMENT`/`SENTRY_RELEASE` from env via a boot-time `instrument.ts`. Sentry now initializes from a `service_configs` row with purpose `sentry` (`{dsn, environment, release}`, empty dsn = disabled, missing row = disabled defaults via `normalizeSentryConfig`), right after the Prisma plugin registers in fastify's `buildApp` and during express's async `start()`. `instrument.ts` and the env vars are gone from both templates; unit tests cover normalization, the empty-dsn no-op, single init, and idempotence. FastAPI and the Vite frontend keep their env-based Sentry config.

### Fixed

- **express template shipped without the `ServiceConfig` Prisma model** ([express/prisma/schema.prisma](express/prisma/schema.prisma)) — `express/src/server.ts` reads the Sentry config from `service_configs` at boot via `getServiceConfig(prisma, SENTRY_PURPOSE)`, but the schema lacked the model, so every scaffolded express + prisma project crashed on start with `prisma.serviceConfig` undefined. The model now mirrors the fastify schema (`purpose String @unique @db.VarChar(64)`, `@@map("service_configs")`); the auth feature's prisma models patch is unaffected (it only inserts auth models), and the `ci-local.sh` express section's boot smoke guards the boot path against Postgres.
- **Free-typed `'smtp'` config purpose replaced by `SMTP_PURPOSE`** ([features/auth/](features/auth/)) — the eight Node auth `03-app-plugins` patches (fastify/express × prisma/drizzle/sequelize/typeorm) each inlined the `'smtp'` literal when wiring `initMailer`. The purpose now lives once per stack in the auth mailer (`features/auth/{fastify,express}/common/files/src/modules/auth/mailer.ts`), is re-exported from the module index, and the patches import it. FastAPI's mailer takes a config-source callable and never typed the literal.
- **express rate limiters keyed raw `req.ip`, tripping express-rate-limit's IPv6 validation** ([express/src/app.ts](express/src/app.ts), [features/auth/express/](features/auth/express/)) — the base app limiter and the drizzle/typeorm auth-port limiters returned `req.ip` from custom `keyGenerator`s, so express-rate-limit 8.x logged `ERR_ERL_KEY_GEN_IPV6` (a ValidationError stack trace) on every boot and IPv6 clients were keyed per-address instead of per-subnet. All keyGenerators now route IPs through the library's `ipKeyGenerator` helper, matching the prisma auth port; a regression test asserts `buildApp` raises no IPv6 keyGenerator validation error.
- **Flat `dist/` layout for backend builds** ([fastify/tsconfig.build.json](fastify/tsconfig.build.json), [express/tsconfig.build.json](express/tsconfig.build.json)) — `tsc` with `include: ["src", "tests"]` emitted `dist/src/**`, breaking `npm start` (`dist/server.js`), the pm2 ecosystem config, and the Dockerfile `CMD`. Builds now run `tsc -p tsconfig.build.json` (`rootDir: "src"`, `include: ["src"]`) while `typecheck` keeps the full tsconfig; the Dockerfile templates and the ORM-addon Dockerfile generator copy `tsconfig.build.json` into the build stage.
- **express + prisma auth tests asserted a non-existent error envelope** ([features/auth/express/prisma/](features/auth/express/prisma/)) — seven tests expected `body.error.code`/`body.error.message`, but the express error handler emits a flat `{detail, code, request_id}`; expectations corrected to the real shape.
- **express template tests realigned with the flat error envelope and auth-by-default routing** ([express/tests/](express/tests/)) — 17 assertions still read `body.error.*` from the old nested envelope, and two `app.test.ts` cases expected 404 for unauthenticated requests to unknown paths even though the global `requireAuth` now answers 401 first; the 404 cases authenticate with a signed test JWT and a new case pins the 401 + `unauthorized` envelope.
- **`bulkDelete` soft-delete tests mocked the wrong error shape** ([fastify/tests/modules/repository.test.ts](fastify/tests/modules/repository.test.ts), [express/tests/modules/repository.test.ts](express/tests/modules/repository.test.ts)) — the repository swallows only Prisma `P2025`, but the mocks rejected with a plain `Error`, so the “skips missing records” cases failed; mocks now carry `code: 'P2025'`.
- **fastify crypto key-resolution tests stubbed a too-short `JWT_SECRET`** ([fastify/tests/lib/crypto.test.ts](fastify/tests/lib/crypto.test.ts)) — config enforces a 32+ char `JWT_SECRET`, so the 20-char stub blew up config load before the assertion under test; the stub is now 48 chars.
- **admin-panel migrate tests no longer hardcode the migration count** ([admin-panel/internal/db/migrate_test.go](admin-panel/internal/db/migrate_test.go)) — the idempotency and concurrency tests derive the expected `schema_migrations` count from the embedded migrations FS, so adding a migration file no longer breaks them.
- **`COOKIE_SECURE` documented** ([admin-panel/.env.example](admin-panel/.env.example)) — the Go config reads it (Secure cookies default on; `false` opts out for plain-HTTP local use) but the example omitted it.
- **Pagination clamp bounds named** ([fastify/](fastify/src/modules/_base/auto-routes.ts), [express/](express/src/modules/_base/auto-routes.ts), [addons/orms/](addons/orms/)) — the inline `10`/`100` page-size literals in the five query normalizers are now `DEFAULT_PAGE_SIZE`/`MAX_PAGE_SIZE` constants.
- **`ci-local.sh` cleanup sweep scoped to the repo** ([scripts/ci-local.sh](scripts/ci-local.sh)) — the EXIT-trap `pkill -f "tsx.*src/server.ts"` matched any `tsx` server on the machine, so a run killed sibling repos' dev backends. The pattern now requires the run's `$ROOT_DIR` prefix, so the sweep only reaps this checkout's boot launchers; the per-PID process-group kill in the `PIDS` loop already covers every server the script starts.
- **Service-config `purpose` constants renamed to the bare `X_PURPOSE` form** ([fastify/src/lib/sentry.ts](fastify/src/lib/sentry.ts), [express/src/lib/sentry.ts](express/src/lib/sentry.ts), [features/auth/](features/auth/)) — `SENTRY_CONFIG_PURPOSE`/`SMTP_CONFIG_PURPOSE` are now `SENTRY_PURPOSE`/`SMTP_PURPOSE` across the Node sentry libs, the auth mailer + module index, and the eight auth `02-app-imports`/`03-app-plugins` patches, matching the agreed nomenclature used across sibling repos. Values are unchanged (`'sentry'`, `'smtp'`).

### Fixed

- **`dist-e2e` excluded from frontend lint and format** ([frontend/eslint.config.js](frontend/eslint.config.js), [frontend/.prettierignore](frontend/.prettierignore)) — the E2E Vite build dir introduced in 1.7.5 was gitignored but not excluded from `eslint .` or `prettier --check .`, so a whole-directory lint (as `scripts/ci-local.sh` and the CI frontend job run) tried to lint the minified bundle and failed with ~1000 `no-undef` errors. Both ignore lists now cover `dist-e2e` alongside `dist`. This unblocks the frontend CI job, which gates publishing.

### Security

- **Infra template hardened to clear all Trivy HIGH/CRITICAL findings** ([infra/stack/](infra/stack/)) — ECR repositories set `image_tag_mutability = "IMMUTABLE"`; the CI/CD artifacts S3 bucket gains a public-access block and SSE-KMS with `bucket_key_enabled`; the compose EC2 host requires IMDSv2 (`http_tokens = "required"`) and encrypts its root volume; RDS security-group egress is scoped to the VPC CIDR; the SNS alerts topic is encrypted. The compose host's internet egress and the SNS AWS-managed key carry documented `#trivy:ignore` justifications (an app host needs arbitrary outbound for image pulls; a billable customer CMK is unwarranted for alarm notifications). Each `terraform` sub-module (`compose/`, `k8s/`) now declares its own `required_version` + `required_providers`, the two deprecated `"${var.x}"` interpolations in `k8s/keycloak.tf` are de-interpolated, and ten unused pass-through variables on the `k8s` module (consumed only by the parent `eks.tf`) are removed from both the module and the call site. `terraform validate`, `tflint --recursive`, and `trivy config` are all clean.
- **Pre-commit hook skips vendored Terraform modules in the Trivy scan** ([.githooks/pre-commit](.githooks/pre-commit)) — `trivy config` now runs with `--skip-dirs .terraform` so downloaded module example manifests (e.g. the EKS module's `examples/*.yaml`) no longer surface findings the template can't fix.

## [1.7.6] - 2026-06-09

### Fixed

- **Unknown component names now report a clear error with a suggestion** ([cli/src/index.ts](cli/src/index.ts), [cli/src/utils.ts](cli/src/utils.ts)) — previously `create-projx add admiin-panel` (or `--components admiin-panel`) silently dropped the unrecognized name and printed the misleading `specify components to add`, as if nothing was passed. Both `add` and `--components` now validate every name and fail with `unknown component admiin-panel — did you mean admin-panel?` (exit code 2, a usage error), using a new `suggestComponent` Levenshtein matcher. Genuinely-empty input still reports `specify components to add`.
- **admin-panel redirect loop behind the frontend nginx proxy** ([frontend/nginx.conf.ejs](frontend/nginx.conf.ejs)) — the `/admin/` reverse-proxy block used `proxy_pass http://$admin_panel_upstream:8055/admin/;` with a variabilized upstream, which (combined with a missing `X-Forwarded-Prefix`) made the Go admin-panel's login route redirect to itself, producing `ERR_TOO_MANY_REDIRECTS` on `/admin/login`. The block now passes through unmodified (`proxy_pass http://$admin_panel_upstream:8055;`, no URI), matches with `^~` so it wins over regex locations, and forwards `X-Forwarded-Prefix /admin` so the app builds correct redirect/cookie paths under its `BASE_PATH`. Test in [cli/tests/admin-panel.test.ts](cli/tests/admin-panel.test.ts) updated to assert the corrected shape.

## [1.7.5] - 2026-06-08

### Changed

- **E2E Playwright suite now actually runs in CI** ([.github/workflows/ci.yml](.github/workflows/ci.yml)) — the `e2e` job previously stopped at format + lint + typecheck + audit and never launched a browser. ### Fixed

- **`pm.exec` rendering as the literal string `undefined` in scaffolded READMEs.** The internal `evalExpr` ([cli/src/utils.ts:551](cli/src/utils.ts#L551)) was rebinding `pm` to the package-manager name string instead of the full `PmCommands` object. Template-literal ternaries that referenced `pm.exec` inside backticks therefore resolved `pm.exec` to `"npm".exec` = `undefined`, producing lines like `... npm install && undefined prisma migrate dev && npm run dev` in every scaffold's quickstart. Simple `pm.install` references used a different fast-path and worked correctly, which is why this slipped through. Fix: pass the full `PmCommands` object into evaluated expressions and update templates to compare `pm.name === 'pnpm'` (etc.) instead of `pm === 'pnpm'`.
- **`scripts/ci-local.sh` was silencing coverage** with `--coverage.enabled=false` for every JS component, so the 80% thresholds declared in each template's `vitest.config.ts` weren't enforced anywhere. Flipped to `vitest run --coverage`. Combined with the express CI change below, ci-local is now the canonical gate before push.
- **Scaffolded express CI step `vitest run --coverage.enabled=false`** removed — it was running tests without enforcing the thresholds. Sibling model picked: scaffolded CI does lint/typecheck/build/audit, ci-local runs tests with coverage pre-push.
- **`sec_e2e` would orphan its backend on Ctrl-C and never resolve `pm_exec`.** `start_background` was called inside a subshell, so the parent's `LAST_PID` / `PIDS` arrays never saw the e2e backend — the script-level cleanup trap couldn't kill it, and even the explicit `kill "$backend_pid"` at end-of-function targeted a stale or empty PID. Separately, `setsid pm_exec tsx …` was passing a shell function (`pm_exec`) to a binary (`setsid`), which couldn't resolve it. Both fixed: `start_background` now runs in the function's own scope (PID propagates), and the daemon command is wrapped as `bash -c "$(declare -f pm_exec detect_pm); pm_exec …"` so `setsid` invokes `bash` (real binary) which then resolves the function. Cleanup also upgraded to process-group kill with TERM→KILL fallback.
- **Pre-commit infra block was `fmt`-only.** Both [.githooks/pre-commit](.githooks/pre-commit) and [cli/src/templates/pre-commit.ejs](cli/src/templates/pre-commit.ejs) now run `terraform validate` + `tflint --recursive` + `trivy config --severity HIGH,CRITICAL` after format, gated on `command -v` so missing binaries don't block commits. `terraform init` is cached with `[ -d .terraform ] || init …` so the hook doesn't re-download providers on every `.tf` edit.
- **`idna` bumped 3.13 → 3.15** (CVE-2026-45409 — DNS label length DoS). Transitive via `anyio` + `httpx` in [fastapi/uv.lock](fastapi/uv.lock). Dependabot PR #49.
